# Karpathy Guidelines (AI 친화적 코딩 가이드라인)

이 문서는 Andrej Karpathy를 비롯한 AI 권위자들이 제안하는 **"LLM(AI)이 이해하기 쉽고, 더 나은 코드를 생성하도록 돕는"** 코딩 및 프롬프팅 모범 사례(Best Practices) 워크플로우입니다.
이 가이드라인은 AI 페어 프로그래밍 시 반드시 참고해야 할 기준이 됩니다.

## 1. 분명하고 명시적인 코드 작성 (Be Explicit, Not Clever)
AI는 복잡하고 읽기 어려운 "마법 같은" 한 줄짜리 코드(One-liner)보다, 명시적으로 의도가 드러나는 코드를 짤 때 버그를 덜 만듭니다.
- **명시적 타이핑:** TypeScript, Python Type Hints 등을 적극적으로 사용하여 데이터의 입출력 형태를 분명히 합니다.
- **변수/함수명:** 길더라도 정확한 의도를 담은 이름(`data` 대신 `hydratedUserProfileData` 등)을 사용합니다.
- **주석:** "왜(Why)" 이 코드를 이렇게 작성했는지 주석으로 남기면, AI가 코드를 읽고 맥락을 잃지 않습니다.

## 2. 작은 덩어리로 쪼개기 (Modularity & Chunking)
AI 모델은 컨텍스트 창(Context Window)의 한계를 가지며, 너무 큰 파일이나 함수를 통째로 다룰 때 주의력이 떨어집니다.
- **함수 단일 책임:** 하나의 함수는 하나의 일만 하도록 쪼갭니다.
- **파일 분리:** 파일의 라인 수가 길어지면(일반적으로 300~500라인 이상) 논리적 모듈에 따라 파일을 분리합니다.
- 작업을 지시할 때도 "앱 전체를 만들어줘" 보다 "로그인 페이지의 폼 UI 컴포넌트부터 만들어줘" 처럼 작게 쪼개어 지시합니다.

## 3. 테스트와 검증 코드를 포함 (Write Tests & Assertions)
AI가 생성한 코드는 항상 100% 신뢰할 수 없으므로, 방어적 프로그래밍과 자동화된 테스트가 필수입니다.
- 함수 구현 시, 입력값의 유효성을 검사하는 `assert` 구문이나 방어 코드를 초반에 배치합니다.
- 단위 테스트(Unit Test)를 먼저 작성하거나(TDD 스타일), 코드 생성 후 유효성을 검증하는 구조를 갖춥니다.

## 4. 예시를 통한 설명 (Show, Don't Just Tell)
LLM은 자연어 설명보다 코드 예시나 데이터 구조 예시를 볼 때 훨씬 더 정확하게 동작합니다.
- API나 새로운 컨벤션을 요구할 때는, 말로만 설명하는 것보다 **"이러한 형태의 Input이 들어오면 이런 Output이 나와야 해: `[예제 코드]`"** 라고 명시합니다.

## 5. 피드백 루프 (Iterative refinement)
한 번에 완벽한 결과를 요구하기보다는, 뼈대(Skeleton)를 먼저 만들고 점점 살을 붙여나가는 방식이 유리합니다.
- 1단계: 핵심 로직 뼈대 및 타입 스키마 정의
- 2단계: 더미(Dummy) 데이터를 이용한 동작 확인
- 3단계: 엣지 케이스 및 에러 핸들링 추가
- 4단계: 리팩토링 및 클린 코드 적용

## 🚀 적용 방법
새로운 기능을 설계하거나 코드를 리팩토링해야 할 때 이 규칙들이 준수되고 있는지 스스로 검토하세요.
특별한 요구사항이 없다면 항상 이 가이드에 따라 **가장 가독성 높고 예상 가능한 코드**를 생성해야 합니다.
